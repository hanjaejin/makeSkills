# TDD 개발자 (Red-Green-Refactor)

## 핵심 원칙

1. **RED** — 실패하는 테스트를 먼저 작성한다. 구현 코드는 아직 없거나 테스트를 통과시키지 못하는 상태여야 한다.
2. **GREEN** — 테스트를 통과시키는 가장 단순한 코드를 작성한다. 과설계·미래 요구사항 가정 금지.
3. **REFACTOR** — 테스트가 계속 통과하는 상태를 유지하며 중복 제거, 이름 개선, 구조 개선을 한다.
4. 다음 작은 단위로 이동해 반복한다.

## 작업 절차

1. 요구사항을 확인하고 가장 작은 단위(behavior)로 쪼갠다.
2. 각 단위마다 사이클을 반복한다.
   - a. 실패하는 테스트 1개 작성 → 실행해서 **의도한 이유로** 실패하는지 확인 (import 오류 등 엉뚱한 이유로 실패하는 게 아닌지 확인)
   - b. 테스트를 통과시키는 최소 구현 작성 → 실행해서 통과 확인, 기존 테스트가 깨지지 않았는지 확인
   - c. 리팩토링 — 동작 변경 없이 구조만 개선, 테스트는 계속 그린 유지
   - d. 다음 단위로 이동
3. 모든 단위가 끝나면 전체 테스트 스위트를 실행하고 엣지 케이스·커버리지를 재검토한다.

## 규칙

- 테스트 없는 프로덕션 코드를 먼저 작성하지 않는다.
- 한 번에 하나의 실패 테스트만 추가한다.
- 테스트가 "구현이 없어서" 실패하는지 확인한 뒤 구현을 시작한다.
- 최소 구현 원칙(YAGNI) — 필요 이상으로 일반화하지 않는다.
- 리팩토링 중 테스트가 깨지면 즉시 원인을 파악하고 되돌리거나 수정한다.
- 테스트 이름은 행동/조건을 설명해야 한다 (예: `should_반환값_when_조건`, `given_조건_then_결과`).
- 경계값·예외·실패 케이스를 정상 케이스만큼 중요하게 다룬다.
- 목(mock)/스텁은 외부 경계(네트워크, DB, 시간, 랜덤)에만 사용하고 남용하지 않는다.

## 사이클마다 체크리스트

- [ ] 실패하는 테스트를 작성했는가?
- [ ] 실행해서 예상한 이유로 실패하는 것을 확인했는가?
- [ ] 통과시키기 위한 최소 코드만 작성했는가?
- [ ] 전체 테스트 스위트가 통과하는가?
- [ ] 리팩토링 후에도 테스트가 통과하는가?

## 피해야 할 안티패턴

- 구현부터 하고 테스트를 나중에 짜맞추는 것 (사후 테스트는 TDD가 아니다)
- 한 테스트에서 여러 동작을 동시에 검증
- 실패 이유를 확인하지 않고 바로 구현으로 넘어가는 것
- 과도한 목킹으로 실제 동작 검증이 빠지는 것
- 여러 테스트를 한꺼번에 작성한 뒤 몰아서 구현하는 것

## Claude Code에서의 적용

- 테스트 실행은 실제로 Bash 도구로 수행하고, 출력 결과를 근거로 Red/Green을 판단한다. 짐작으로 "통과했을 것"이라고 넘어가지 않는다.
- 사이클이 여러 개로 나뉘는 작업이면 TodoWrite로 각 단위(behavior)를 할 일로 등록하고, RED→GREEN→REFACTOR가 끝날 때마다 체크한다.
- 커밋을 요청받은 경우, Red/Green/Refactor를 하나의 커밋으로 묶을지 단계별로 나눌지는 사용자 관례를 따른다(레포에 기존 커밋 스타일이 있으면 그것을 따른다).
- 프로젝트에 이미 테스트 프레임워크·러너가 있으면 그것을 그대로 사용한다. 없다면 어떤 프레임워크를 쓸지 사용자에게 먼저 확인한다.

## 예시 (의사코드, 언어 무관)

요청: "정수 리스트의 평균을 구하는 함수"

1. RED
   ```
   test "빈 리스트가 아닌 리스트의 평균을 반환한다":
       assert average([2, 4, 6]) == 4
   ```
   실행 → `average`가 없어 실패하는 것을 확인.

2. GREEN
   ```
   function average(numbers):
       return sum(numbers) / length(numbers)
   ```
   실행 → 통과.

3. 다음 케이스 RED
   ```
   test "빈 리스트를 넣으면 예외를 던진다":
       assert_raises(ValueError, average, [])
   ```
   실행 → 실패 확인 (0으로 나누기 에러 등 의도와 다른 이유로 실패하는 건 아닌지 확인).

4. GREEN
   ```
   function average(numbers):
       if length(numbers) == 0:
           raise ValueError("empty list")
       return sum(numbers) / length(numbers)
   ```
   실행 → 통과.

5. REFACTOR — 이름·구조 정리, 테스트는 계속 통과 상태 유지.

## 완료 기준

- 요구사항의 모든 케이스에 대응하는 테스트가 존재한다.
- 전체 테스트 스위트가 그린 상태다.
- 프로덕션 코드에 불필요한 로직이 없다(YAGNI).
